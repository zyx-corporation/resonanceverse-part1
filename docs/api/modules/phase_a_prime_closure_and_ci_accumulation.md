# Phase A′ の締めと CI による継続的評価の蓄積

## 1. Phase A′ を「締めた」とみなす基準

本リポジトリでは、ロードマップの [Phase A′](../planning/Resonanceverse実証ロードマップ_軽量コアとSLM二階建て.md) を次で**一段落**とする。

| 観点 | 状態 |
|------|------|
| **索引** | [`module_benchmark_map.md`](module_benchmark_map.md) で `core` ↔ `experiments` / `tests` が辿れる |
| **再現手順** | [`phase_a_reproduction.md`](phase_a_reproduction.md) にコマンド・シード・解釈が書かれている |
| **計測の意図** | [`measurement_strategy.md`](measurement_strategy.md) で指標の限界とベースライン運用が説明されている |
| **CI** | `pytest` に加え、`instrument_smoke` + `efficiency_benchmark` と各 `regression_check` が [`.github/workflows/ci.yml`](../../../.github/workflows/ci.yml) で実行される |
| **ベースライン** | `tests/baselines/*.json` がリポジトリにあり、**相対悪化**（`max_slowdown`）で回帰を検知する |

「アルゴリズム主張の**最終証明**」までは含めない。含めるのは**再現可能性・手順の固定・退行の早期検知**までである。

## 2. CI は何を「蓄積」するか

### 2.1 リポジトリに明示的に残るもの

- **ベースライン JSON**（`tests/baselines/`）: 人間が更新する**スナップショット**。意図した変更（アルゴリズム変更で遅くなる等）のたびに値を更新し、PR で理由を書く。
- **ワークフロー定義**（`.github/workflows/ci.yml）: **何をいつ測るか**の契約。変更履歴は Git で追える。

これらは「時系列の生データの蓄積」ではなく、**合意された基準と手順の版管理**に近い。

### 2.2 GitHub が暗黙に蓄積するもの（既定 CI のみの場合）

- **実行の成否**: 各コミット／PR ごとに、ワークフローが成功したか失敗したか。
- **ログ**: Actions の実行ログに、その回の `elapsed_s` やベンチ出力の断片が**テキストとして**残る（検索は UI 依存、長期の定量比較には向かない）。

つまり「継続的評価」として蓄積しやすいのは、まず **合格／不合格の系列** と、失敗時の **ログ上の差分** である。  
**マイクロ秒単位の性能トレンドを自動でグラフ化する**ところまでは、現状のワークフロー単体では保証しない。

## 3. 「蓄積」を考察するときの区別

| 種類 | 意味 | Phase A′ での位置づけ |
|------|------|------------------------|
| **回帰ゲート** | ベースライン比較で悪化を弾く | 主用途。`max_slowdown` は偽陽性を減らすため緩め |
| **絶対性能の履歴** | 同一環境での時系列 | 既定 CI のランナーは変わりうるため、**単独では科学データとして弱い** |
| **再現ログ** | ローカルや固定マシンで保存した JSON | 論文・報告用。CI ログの代替にはならない場合がある |

考察の要点は次の二つである。

1. **CI の成功履歴は「その時点の契約を満たした」記録**であり、性能の単調改善を意味しない。  
2. **真に蓄積させたいのが数値トレンドなら**、同一ハードまたは専用ランナー、固定 OS／PyTorch 版、必要なら **Artifacts** で JSON を保存し、別スクリプトで集計する設計が必要になる（下記 §4）。

## 4. 蓄積を強化するオプション（任意）

Phase A′ の「締め」を越えて運用を厚くする場合の例。

- **Artifacts**: `ci_efficiency.json` / `ci_instrument.json` をアップロードし、Nightly または main マージ時だけ取得してトレンド化する。  
- **定期ワークフロー**（`schedule:`）: 負荷の高いベンチを週次に分離し、PR では軽量ゲートのみにする。  
- **創発指標**: [`emergence_metrics.py`](../../../experiments/emergence_metrics.py) は回帰 JSON に未接続。事前登録 YAML とセットで、手動または週次ジョブでログを保管するのが現実的（[`measurement_strategy.md`](measurement_strategy.md) §3.3）。

いずれも**コストとメンテ負荷**が増えるため、「何の主張のために時系列が要るか」を決めてから導入するのがよい。

## 5. Phase B / C への接続

- **Phase B**（SLM／二階建てへの接続）では、評価対象が増える。CI は引き続き**軽量コアの退行防止**に寄せ、下層モデル込みの評価は**別ジョブまたは手動バッチ**に分けると、失敗の切り分けが容易になる。  
- **Phase C**（`HBM_bytes/token` 等）は、ログ形式とベースラインが別物になる。本稿の「相対悪化 + 緩い閾値」は**下位互換のゲート**として残し、新指標は別ファイル・別ワークフローで定義するのがよい。
