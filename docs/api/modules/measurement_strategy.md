# 計測戦略の考察（Phase A / A′）

## 1. 目的とレイヤ

| レイヤ | 何を知りたいか | 主な手段 |
|--------|----------------|----------|
| **マクロ** | 系列長・ノード数スケールで、全対全と ROI のどちらが有利か | `efficiency_benchmark.py`（`mean_time_s`、CUDA 時 `cuda_peak_bytes`） |
| **ミクロ** | 前向きパス内のどの区間が支配的か | `StageTimer`（区間 `elapsed_s`、CUDA 時区間 `cuda_allocated_delta_bytes`） |
| **回帰** | リファクタや依存更新で**意図しない遅延化**がないか | `regression_check.py`（ベースライン JSON × `max_slowdown`） |

マクロは「設計上の計算量主張」と対応しやすい。ミクロはボトルネック特定とプロファイラ接続の前段に使う。

## 2. 指標の意味と限界

### 2.1 壁時計時間（`elapsed_s` / `mean_time_s`）

- **利点**: 実装・ハード全体を通した実効値。
- **限界**: CPU では OS スケジューラ・キャッシュ・同時実行の影響で**ばらつきが大きい**。同一マシンでも連続実行で 2 倍近く変動しうる。
- **対策**: 複数回反復（`repeats`）、ベースライン比較では **`max_slowdown` を緩めに**（例: 80〜100）。「絶対閾値」より**相対悪化**検出を主用途にする。

### 2.2 CUDA `memory_allocated` 差分（区間ごと）

- **利点**: どの演算ブロックが**追加割り当て**を誘発したかの手掛かり。
- **限界**: `synchronize` 前後の差なので、プール再利用で 0 に見えることがある。区間内のピークは `StageTimer` が CUDA 時に区間開始で `reset_peak_memory_stats` したうえで `max_memory_allocated` を `cuda_peak_allocated_bytes` として記録する（連続区間前提・ネスト時は解釈に注意）。
- **Phase C** では二階建て計画の **`HBM_bytes/token`** など、よりレイヤ寄りの設計指標が主役になる。

### 2.3 再現性

- `set_experiment_seed` で Python / NumPy / PyTorch の乱数は揃える。
- GPU カーネルの非決定性は残るため、**数値完全一致**は期待しない。回帰は**時間・メモリのオーダー**にフォーカスする。

## 3. ベースライン比較の設計

### 3.1 効率ベンチ（`efficiency` モード）

- 比較対象: 各 `seq_len` について `full_quadratic.mean_time_s` と `roi_tiers.mean_time_s`。
- 判定: `現在 > ベースライン × max_slowdown` なら失敗（**悪化のみ**）。現在がベースラインより速い場合は常に合格。
- ベースライン更新: 代表環境で `efficiency_benchmark` を再取得し、`tests/baselines/efficiency_cpu_short.json` を更新。アルゴリズム変更で**正当に遅くなった**場合は `max_slowdown` ではなくベースライン値自体を見直す。

### 3.2 計測フック（`instrument` モード）

- 比較対象: `instrument_smoke.py` 出力の `stages[]` を名前で対応付け、各 `elapsed_s`。
- 判定: 同上（区間ごと）。
- **区間名**は実装と同期させる（リネーム時はベースラインと `regression_check` の両方を更新）。

### 3.3 なし／保留

- **創発指標**（KL 等）: 分布の変化が主で、単一スカラの「悪化」は解釈が難しい。別途、事前登録した閾値との比較が適する。
- **ResonantCore の StageTimer**: 回帰用 JSON 出力は未同梱。必要なら `instrument_smoke` と同様の `--out` スクリプトを追加可能。

## 4. CI での運用

- **緩い `max_slowdown`**: ランナー性能差を吸収（効率 80、計測 60〜100 など）。
- **偽陽性が続く場合**: ベースラインを**そのランナー代表値**に更新するか、`max_slowdown` を上げる（いずれも PR で理由を明記）。
- **偽陰性**: 閾値が緩すぎると大きな退行を見逃す。定期的にローカルや専用ベンチで**絶対値**を確認する。

## 5. 今後（Phase C）

- `HBM_bytes/token`、p50/p95 レイテンシ、レイヤ別 IO は二階建て実装計画に沿い、**別ベースラインとログ形式**を導入する。本稿の CPU ミクロ計測はその**下位互換**として位置づけられる。
