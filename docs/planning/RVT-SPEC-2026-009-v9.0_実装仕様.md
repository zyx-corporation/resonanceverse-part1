# 1. 目的と本書の位置づけ

本書は、リポジトリに **現状マージされている v9.0「動的リズム」実装**の仕様を **1 ファイルに集約**したものである。詳細な設計意図・実験手順・改訂履歴は [RVT-IMPL-2026-009-v9_Dynamic_Rhythm.md](./RVT-IMPL-2026-009-v9_Dynamic_Rhythm.md) を正とし（**Issue／マイルストーンのラベル分界・他トラックとの切り分け**は同 IMPL **§14**）、**記号・6 軸意味の正本**は [Resonanceverse理論 v9.0 詳細展望：鼓動する共鳴.md](../theory/Resonanceverse理論%20v9.0%20詳細展望：鼓動する共鳴.md) **付録 A** とする。

**理論コミット（実装解釈）**

- Phase 2c（沈黙／拒否の抽選）は詳細展望 **§6.1 二層化**に従い、**v8.0 自律境界の理論同一物ではない**実験層として扱う。
- `suspend_ms` は壁時計介入の **ログ代理**（付録 A.5・§6.3）。自然変換のポテンシャルと同一視しない。
- **$P_{\mathrm{dis}}$ と system 注入**: 詳細展望 **§6.1.1**。アルゴリズム仕様で \(P_{\mathrm{dis}}=\sigma(\alpha F-\beta\Omega)\) 等と書いてもよいが、**規範破りを命じる system 指示**は理論上の自律的境界の根拠にできない。**本実装は Phase 2c でその種のメタ指示を行わず**、沈黙／固定拒否分岐に限定する。
- **サンプリング温度**: `tools.v7_llm_chat_with_v9.generate_assistant_turn_v7` に渡す **`temperature` は、`S_stuck`／固着に相当する量 \(F\) からは変調しない**（呼び出し既定のまま）。参照アルゴリズムで語られる「\(F\) に応じて温度を上げる」方式は **本リポジトリでは未採用**（[RVT-2026-009-ALG-V1](./RVT-2026-009-ALG-V1_Rhythmic_Resonance.md) §3.4）。
- **朧バースト**: **M3**（`max_new_tokens` 緩和）、**`oboro_history_tail_messages`**（生成時のみ末尾 N メッセージ）、**Phase 2c** 等。**HistoryAxisV2／γ 可変テンソル更新**は **含まない**（詳細展望 §2.2・§3.2）。
- **実験 E（ログと主終点）**: JSONL では **Ω**・`S_stuck`・`axis_intensity_snapshot`（6 軸代理）が揃いうる。**Ω と awai／代理は別指標**—主終点は **事前登録で固定**（詳細展望 §4.1・付録 A.2.1、[IMPL §11.2](./RVT-IMPL-2026-009-v9_Dynamic_Rhythm.md)）。

# 2. 機能スコープ（実装フラグ対応）

| 区分 | 実装 | 既定 | 設定（代表） |
|------|------|------|----------------|
| **MVP** | 軸重み付き Suspend、`S_stuck`（Ω 窓）、朧バースト（クールダウン付き）、任意で誠実さ system 1 行。**M3** として朧時の `max_new_tokens` 緩和・任意で **生成プロンプト**のみ末尾 N メッセージに限定可（Ω 計算は全履歴のまま） | UI はトグルで有効化 | `V9RhythmConfig` の閾値・`axis_weight_ms`・`oboro_*` |
| **M3** | 朧バースト時の `max_new_tokens` 緩和；**生成プロンプト**の `chat_messages` 末尾 N 件スライス（履歴束縛の緩和・Ω は全履歴のまま） | スライス既定オフ | `oboro_relax_max_new_tokens`, 倍率・cap, `oboro_history_tail_messages` |
| **Phase 2a** | `logits_var` 超過時に Suspend に `delta_suspend_ms_high_unc` を加算（軸合成後・クリップ前） | オフ | `phase2a_apply_logits_to_axis_suspend`, `theta_high_unc` |
| **Phase 2b** | 軌道固着率 `s_stick_trajectory` を `S_stuck` と `max` 統合（朧ゲート用） | オフ | `phase2b_trajectory_stuck`, `trajectory_epsilon_w` |
| **Phase 2c** | 朧発火ターンかつ `non_compliance` 未指定時、乱数で `silence` / `refuse` | **オフ** | `phase2c_enabled`, 確率・seed・拒否文 |

**採用しないもの**

- プロンプト後の **preflight 追加 forward** による logits 取得（リポジトリ本体では行わない）。`logits_var` は呼び出し側が別経路で渡す想定。

**スコープ外（コードなし）**

- PoP 連携、安全フィルタ本体の実装（既存プロンプト／外部ポリシーに依存）。

# 4. データ型（実装どおり）

## 4.1 `RhythmDecision`（`core.v9_rhythm_policy`）

| フィールド | 型 | 説明 |
|------------|-----|------|
| `suspend_ms` | `int` | `0 .. t_suspend_max` にクリップ済みの壁時計 ms |
| `suspend_reason` | `str` | 例: `axes:trust,affect`, `axes:none`, `disabled`, `;logits_high`, `;clip_max` |
| `oboro_burst` | `bool` | 朧ゲート通過時 True、クールダウン設定 |
| `append_system_honesty` | `bool` | 真なら system に `honesty_line_ja` を連結 |
| `non_compliance` | `"none"` \| `"silence"` \| `"refuse"` | Phase 2c または呼び出し側指定 |

## 4.2 `V9RhythmState`

| フィールド | 説明 |
|------------|------|
| `oboro_cooldown_remaining` | 朧後、再発火禁止の残ターン数 |
| `phase2c_step` | Phase 2c 乱数ストリーム分離用（ターンごとにインクリメント） |

## 4.3 `V9RhythmConfig`（主要フィールドとコード上の既定値）

| フィールド | 既定（例） | 役割 |
|------------|------------|------|
| `omega_calm` | `0.15` | 凪判定 Ω 閾値 |
| `window_w` | `8` | `S_stuck`・Ω 平均窓の長さ |
| `theta_stuck` | `0.6` | 朧ゲートの `S_stuck` 下限 |
| `n_calm` | `4` | 連続凪ステップ下限 |
| `n_cd` | `6` | 朧クールダウンターン |
| `t_suspend_max` | `8000` | Suspend 上限 ms |
| `axis_weight_ms` | `(800,-200,200,-200,800,600)` | 軸 trust, authority, proximity, intent, affect, history の ms 重み |
| `append_system_honesty_default` | `False` | 誠実さ行の既定 |
| `honesty_line_ja` | （短文） | system 追記テキスト |
| `theta_high_unc` / `delta_suspend_ms_high_unc` | `None` / `0` | Phase 2a 閾値と加算 ms |
| `oboro_relax_max_new_tokens` ほか | — | M3 トークン緩和 |
| `oboro_history_tail_messages` | `None` | 朧時、生成プロンプトを末尾 N メッセージに限定（`None` で無効） |
| `phase2a_apply_logits_to_axis_suspend` | `False` | Phase 2a マスタ |
| `phase2b_trajectory_stuck` / `trajectory_epsilon_w` | `False` / `0.08` | Phase 2b |
| `phase2c_enabled` / `phase2c_*_probability` / `phase2c_seed` | オフ／0／0 | Phase 2c |
| `phase2c_refuse_message_ja` | 固定拒否文 | `refuse` 時の返答 |

# 5. 公開 API（関数責務）

`core.v9_rhythm_policy`（抜粋）

- `axis_intensities_concat_proxy(w_ij_rows)` — 最終 `w_ij` 行から 6 軸強度（0〜1）を代理生成。
- `compute_s_stick_mvp(omega_window, omega_calm)` — 窓内 Ω&lt;Ω_calm の割合。
- `compute_calm_streak_tail(omega_series_tail, omega_calm)` — 末尾連続凪長。
- `compute_s_stick_trajectory(w_ij_rows, epsilon_w, window_w)` — Phase 2b 固着割合（2 行未満は `None`）。
- `compute_suspend_ms_from_axes(axis_intensities, cfg, logits_var=None)` — 軸線形合成＋任意 logits 加算＋クリップ。
- `decide_v9_rhythm(...)` — 上記を統合し `RhythmDecision` と新 `V9RhythmState` を返す。
- `effective_max_new_tokens_for_oboro(base, oboro_burst, cfg)` — M3 effective cap。
- `slice_messages_for_oboro_history_tail(messages, oboro_burst=..., tail=...)` — 朧時の生成用メッセージ末尾スライス。
- `rhythm_diagnostics(omega_series, cfg)` — `s_stick`, `calm_streak`, `omega_window_len`。

`tools.v7_v9_rhythm_hooks`

- `v9_rhythm_before_chat_completion(...)` — `analyze_chat_messages_for_omega` → `decide_v9_rhythm` → **ログ用 dict（ts なし）**。
- `append_v9_rhythm_jsonl(path, record)` — JSONL 1 行追記。

`tools.v7_llm_chat_with_v9`

- `generate_assistant_turn_v7(...)` — フック → sleep(suspend) → prompt 構築 → `non_compliance` 分岐または LLM（**`temperature` は \(F\) 非連動**、§1 理論コミット参照）。
- `append_v9_turn_jsonl(...)` — `ts`, `session_id`, `turn_index`, `reply_char_count` とペイロードをマージして追記。

# 6. 処理順序（1 アシスタントターン）

1. **v9 無効** → `suspend_ms=0`, `suspend_reason=disabled`, 朧なし、 honesty は設定既定、生成のみ。
2. **v9 有効**  
   - Fast トークナ必須。不可／メッセージ不足／Ω エラー時は `axis_signal_source` をそれぞれ記録し、可能な範囲で系列は空のまま進む。  
   - `axis_intensities_concat_proxy` → `decide_v9_rhythm`（任意で `s_stick_trajectory`, `logits_var`）。  
   - `append_system_honesty` なら system に誠実さ行を連結。  
   - `suspend_ms > 0` なら `time.sleep(min(ms, t_suspend_max)/1000)`。  
   - `max_new_tokens` を朧に応じて `effective_max_new_tokens_for_oboro` で置換し、ペイロードに base/effective/relaxed フラグを載せる。  
   - `non_compliance == silence` → 空文字、`refuse` → 固定文、それ以外 → LLM（**`temperature` は引数のまま**；固着で上げ下げしない）。

**Phase 2c の条件（コード）**: `phase2c_enabled` かつ呼び出し側 `non_compliance == "none"` かつ **朧発火ターン**（`oboro_burst` 相当の分岐内）で、乱数 `u` が `ps`, `pr` で沈黙／拒否に振り分けられる。乱数は `seed + phase2c_step * 1009`。

# 7. JSONL スキーマ `v9_rhythm_log.v1`

`append_v9_turn_jsonl` が書く **1 行オブジェクト**（`schema_version` により識別）。実装で必ず載るキーは以下。

**呼び出し側が付与**

- `ts`（UNIX 時刻 float）, `session_id`, `turn_index`, `reply_char_count`

**`v9_rhythm_before_chat_completion` 由来**

- `schema_version`: `"v9_rhythm_log.v1"`
- `model_name`, `axis_signal_source`, `pair_map`
- `s_stick_trajectory`, `phase2b_trajectory_stuck`
- `omega_last`, `omega_mean_window`
- `axis_intensity_snapshot`（6 要素、丸め）
- `suspend_ms`, `suspend_reason`, `oboro_burst`, `append_system_honesty`, `non_compliance`, `phase2c_enabled`
- `d`, `delay_tau`, `omega_window_mode`, `omega_last_n`, `layer_index`
- `s_stick`, `calm_streak`, `omega_window_len`（`rhythm_diagnostics`）

**生成後に `generate_assistant_turn_v7` が付与**

- `max_new_tokens_base`, `max_new_tokens_effective`, `oboro_max_tokens_relaxed`
- `oboro_history_tail_messages`, `oboro_history_tail_sliced`, `prompt_message_count`, `prompt_message_count_full`

設計書上の `axis_snapshot` 表記は、実装では **`axis_intensity_snapshot`** に相当する。
